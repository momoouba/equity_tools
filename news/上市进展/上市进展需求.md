上市进展管理系统需求文档（开发交付版）

一、文档概述

本文档明确上市进展管理系统的核心需求、数据表设计、业务逻辑、前端页面及验收标准，用于开发落地、测试评审、上线验收，全程遵循系统现有组件复用原则，确保功能闭环、可执行性强。

核心范围：三大交易所 IPO 进展爬取与存储、底层项目管理、业务数据库 SQL 同步、项目与进展匹配、权限与菜单、**系统配置中的上市数据配置（含 Cron）**、邮件与收件管理、日志与重试策略对齐新闻模块；**工程代码按「业绩看板应用」模式独立目录存放，文件夹统一命名为「上市进展」**（见第十五节）。

二、权限、管理员与菜单

1. **第三条顶层分组「上市进展」**：权限位类似 **`hasListingPermission`**（命名以实现为准）。
2. **管理员判定（写死）**：系统用户 **`users.account === 'admin'`** 的账号为管理员，享有文档中全部「仅 admin」能力。
3. **普通用户**：须经授权具备「上市进展」菜单权限后方可访问对应接口与页面；**无上市进展权限的用户不得创建上市进展相关的收件规则**（见「收件与邮件」）。
4. **数据访问**：核心业务表**不设计业务 `app_id` 字段**；行级数据结合**菜单权限**与 **创建人 / 管理员** 规则鉴权（见「接口约定」）。

**定时更新（业务库 SQL）**

- 用户选择 **业务数据库连接**（见「数据库连接」），编写 **只读 SQL**，结果列与 `ipo_project` 业务字段做 **映射**（**不含** 以 `f_` 开头的系统字段）。
- 流程：**SQL 结果先进入临时存储 → 与当前库对比 → 再执行追加或更新**。
- **去重键**：**归属基金 + 归属子基金 + 企业全称**（与旧版「基金+子基金+项目简称」不同，以本澄清为准）。

- 连接信息使用系统已有 **`external_db_config`**，**每用户一套**，交互与 **新闻舆情** 菜单中「数据库连接配置」一致，**直接复用调用**。

2. 其他资源的创建人规则仍以文档及现有中间件为准。

六、收件与邮件

1. **收件配置表**：使用 **`recipient_management`** 存储收件人等内容；该表 **新增字段 `app_id`**（VARCHAR，对应 `applications.id`）：
   - **新闻舆情** 场景：存 **新闻舆情** 对应应用的 `app_id`；
   - **上市进展** 场景：存 **上市进展** 对应应用的 `app_id`。
2. **`applications` 表初始化（或迁移）一条应用记录（写死逻辑）**：
   - **id**：`2026033000000000001`
   - **名称**：`上市进展`
   - **创建时间**：`2026-03-30 18:00:00`（若表结构仅有名称等字段，以实现为准，**id 与名称不得冲突**）。
3. **`email_config`**：**必须存在「上市进展」应用** 对应的邮件配置记录，**用于上市进展发信**；**与新闻舆情应用配置分离**，**逻辑写死**：发上市进展类邮件时 **只解析并使用 `app_id = 上市进展应用 id` 的 `email_config`**。
4. **发送触发**：
   - **新闻**相关邮件：走新闻侧收件配置 + 新闻应用 `email_config`；
   - **上市进展**相关邮件：走上市进展收件配置（`recipient_management.app_id = 上市进展应用`）+ **上市进展应用** `email_config`。
5. **权限**：用户须在 **上市进展 Tab2 收件管理** 中 **创建过自己的上市进展收件信息**，才可能收到上市进展邮件；**无上市进展菜单权限的用户不能创建该类收件规则**。

---

七、邮件功能说明（内容与时点）

1. **SMTP**：通过 **`email_config`**，且上市进展 **固定绑定上市进展应用**（见上节）。
2. **日报内容**：邮件 **不分详情链接**；正文分 **两部分**：**① 底层项目上市进展**；**② 最新获取的上市企业进展**（结构以实现为准）。
3. **Tab2「发送邮件」**：**不选择日期范围**；**仅**在 **定时任务 / 爬虫按配置取数** 等流程中才使用 **时间范围**（与新闻侧「取数+发信」一致处对齐）。
4. **日志保留时长、邮件发送失败重试**：与 **新闻模块一致**（复用同一套运维与代码策略，以实现为准）。

---

八、定时爬取、节假日与匹配批次

1. **每日自动爬取** 由 **系统配置 → 上市数据配置** 中的 **Cron 表达式** 定义 **北京时间** 触发时刻；到点自动执行 **爬虫或数据接口** 配置。
2. **抓取时间窗**：默认定时任务 **抓取「前一日」** 上市进展（与第七节 SQL 中「前一日」为 **同一套语义**）。
3. **节假日**：若定时任务配置为 **跳过节假日**，则本次执行应 **补抓「节假日前一自然日」至「当前执行日的前一自然日」** 区间内尚未入库的数据（具体边界以实现与 Cron 库行为为准，需写测试用例）。
4. **同日多次触发**：对已抓取内容 **追加尚未写入数据库的数据**（幂等：不重复插入已存在主键或业务唯一键，以实现为准）。
5. **匹配时 SQL 语义示例**（与「前一日」一致）：匹配操作可选用类似  
   `WHERE DATE(ip.f_update_time) = DATE_SUB(CURDATE(), INTERVAL 1 DAY)`  
   表示 **进展侧更新日期为北京时间「昨日」** 的批次参与匹配（与手动发信 **无范围** 不冲突）。

---

九、匹配规则与笛卡尔展开

1. **括号处理（澄清后规则）**：对 **`ipo_progress.company` 与 `ipo_project.company` 分别** 做 **去括号处理后再比较**。  
   - **规则**：去掉 **全角、半角括号字符**，**括号内文字保留** 并直接拼接在外层文字中（与仅删除括弧、保留内部文字的常见实现一致）。  
   - **示例**：交易所侧 `华太电子（深圳）有限公司` → `华太电子深圳有限公司`；底层项目侧 `华太电子(深圳)有限公司` → `华太电子深圳有限公司`；二者相等则匹配。
2. **同一企业、同一自然日、多交易所多条 `ipo_progress`**：与底层项目 **LEFT JOIN / 组合** 时，**不做合并**；若底层项目 **4 条**、同日 **2 条** 交易所进展，则 **`ipo_project_progress` 期望为 8 条**（**4×2**）。实现上以保证 **全量组合** 为目标。

十、历史与日期策略（北京时间）

「昨日 / 本周 / 本月 / 全部」及爬取、匹配中的「前一日」均以 **中国北京时间** 日历日为准。

十二、异常与通知

1. **抓取失败**：写明确日志；**失败告警邮件** 收件人策略与前期需求一致：**非业务日报对象**（如仅 admin 或单独告警配置），与新闻模块 **对齐实现**。
2. **抓取成功但无数据**：业务日报仍发「今日未抓取到上市数据」类说明；收件人为 **上市进展收件配置** 用户。

**项目编号（`project_no`）生成规则**

- **格式**：`PP` + **年月日（YYYYMMDD，8 位）** + **5 位序号**（自 `00001` 起左侧补零），示例：`PP2026033000001`。
- **生成逻辑**：与 `invested_enterprises` 表中 **`project_number`** 的实现一致（参见 `server/routes/enterprises.js` 中 `generateProjectNumber`）：以 **`PP` + 当前日期的 `YYYYMMDD`** 为前缀，在 **`ipo_project` 表** 中查询当日已存在的最大 `project_no`，序号 **+1**；若无则序号为 `1`；最终拼接为 **前缀 + 5 位序号**。
- **与舆情被投的差异**：舆情被投为前缀 **`P`**，上市进展底层项目为前缀 **`PP`**，其余（按日滚动序号、5 位宽度、同日递增）**同一套算法**。

**去重（业务库 SQL 定时更新）**：**归属基金 + 归属子基金 + 企业全称**；命中后若 **剩余金额、穿透权益占比、穿透剩余金额** 与库内不一致则 **更新** 业务列，**项目编号不变**，并刷新 **biz_update_time**。

**批量导入去重**：若与 SQL 同步键不同，以实现与模板约定为准（可单独配置为 **基金 + 子基金 + 项目简称** 等，需与产品一致）。

十五、工程目录与代码组织（对齐「业绩看板应用」）

上市进展为**相对独立**的业务域时，**不得**将相关代码零散堆在通用目录；须与 **「业绩看板应用」**（`业绩看板应用` 文件夹）采用**同一套隔离方式**：**单独建目录、分模块存放**，便于维护与边界清晰。

1. **目录命名（固定）**：统一使用 **`上市进展`** 作为文件夹名称（与顶层菜单名称一致）。
2. **参考范例**：`news/业绩看板应用/`、`news/server/routes/业绩看板应用/`、`news/client/src/pages/业绩看板应用/`、`news/client/src/components/业绩看板应用/`、`news/client/src/api/业绩看板应用/` 等。
3. **上市进展落地时建议对应关系（实现时按实际拆分）**：
   - **仓库内文档 / SQL / 附属脚本**：`news/上市进展/`（本需求文档已位于此路径下，可继续在此放置 `docs`、迁移 SQL 等）。
   - **后端路由与业务**：`news/server/routes/上市进展/`（入口 `index.js` 再分子模块，风格对齐 `业绩看板应用`）。
   - **前端页面**：`news/client/src/pages/上市进展/`。
   - **前端可复用组件**：`news/client/src/components/上市进展/`（若有）。
   - **前端 API 封装**：`news/client/src/api/上市进展/`（若有）。
4. **原则**：仅与上市进展强相关的页面、接口、爬虫/任务脚本、专用工具类放入上述目录；**公共能力**（如 `db.js`、`email_config`、通用中间件）仍走系统公共模块，通过引用接入。
