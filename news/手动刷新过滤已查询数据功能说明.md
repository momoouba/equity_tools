# 手动刷新过滤已查询数据功能说明

## 📋 功能概述

当一天内多次手动刷新企查查或新榜接口时，系统会自动过滤掉当天已经查询过的公众号ID（新榜）和企业信息（企查查），只查询未查询过的数据，避免重复查询和浪费API调用次数。

## 🔧 实现逻辑

### 新榜接口过滤逻辑

**位置：** `news/server/routes/news.js` - `executeNewsSyncForConfig` 函数

**过滤条件：**
- 仅在手动触发时生效（`isManual = true`）
- 查询当天（基于Asia/Shanghai时区）已同步的新闻数据
- 从 `news_detail` 表中获取已查询的公众号ID（`wechat_account` 字段）
- 过滤掉已查询的公众号ID，只查询未查询过的

**实现步骤：**
1. 获取当天开始和结束时间（Asia/Shanghai时区）
2. 查询 `news_detail` 表中当天已同步的公众号ID（`APItype = '新榜'`）
3. 从待查询的公众号ID列表中过滤掉已查询的
4. 如果所有公众号都已查询过，返回提示信息

**代码示例：**
```javascript
// 如果是手动触发，过滤掉当天已查询过的公众号ID
if (isManual) {
  // 获取当天开始时间（Asia/Shanghai时区）
  const todayStart = new Date(localYear, localMonth - 1, localDay, 0, 0, 0);
  const todayEnd = new Date(todayStart);
  todayEnd.setDate(todayEnd.getDate() + 1);
  
  // 查询当天已同步的公众号ID
  const syncedAccounts = await db.query(
    `SELECT DISTINCT wechat_account 
     FROM news_detail 
     WHERE APItype = '新榜' 
     AND created_at >= ? AND created_at <= ?
     AND wechat_account IS NOT NULL 
     AND wechat_account != ''`,
    [todayStart, todayEnd]
  );
  
  // 过滤掉已查询的公众号ID
  uniqueAccounts = uniqueAccounts.filter(account => !syncedAccountSet.has(account));
}
```

### 企查查接口过滤逻辑

**位置：** `news/server/routes/news.js` - `syncQichachaNewsData` 函数

**过滤条件：**
- 仅在手动触发时生效（通过 `logId` 判断，手动触发会创建 `logId`）
- 查询当天（基于Asia/Shanghai时区）已同步的新闻数据
- 从 `news_detail` 表中获取已查询的企业全称（`enterprise_full_name` 字段）
- 通过企业全称映射到统一社会信用代码，过滤掉已查询的企业

**实现步骤：**
1. 获取当天开始和结束时间（Asia/Shanghai时区）
2. 查询 `news_detail` 表中当天已同步的企业全称（`APItype = '企查查'`）
3. 创建企业全称到统一社会信用代码的映射
4. 找到已查询企业的统一社会信用代码
5. 从待查询的统一社会信用代码列表中过滤掉已查询的
6. 如果所有企业都已查询过，返回提示信息

**代码示例：**

## 📊 功能特点

### 1. 时区处理
- 使用 Asia/Shanghai 时区计算当天时间范围
- 确保跨时区环境下的准确性

### 2. 错误处理
- 如果过滤过程出错，不会中断同步流程
- 继续使用所有数据源，确保功能可用性

### 3. 日志输出
- 记录过滤掉的公众号/企业数量
- 记录剩余待查询的数量
- 便于监控和调试

### 4. 智能判断
- **新榜**：通过 `isManual` 参数判断是否为手动触发
- **企查查**：通过 `logId` 参数判断是否为手动触发（手动触发会创建日志记录）

## 🎯 使用场景

### 场景1：多次手动刷新新榜接口

**第一次刷新：**
- 查询所有公众号ID（例如：100个）
- 同步数据并存储到 `news_detail` 表

**第二次刷新（同一天）：**
- 查询当天已同步的公众号ID
- 过滤掉已查询的（例如：过滤掉80个）
- 只查询剩余的公众号ID（例如：20个）

**第三次刷新（同一天）：**
- 如果所有公众号都已查询过，返回提示："所有公众号今天都已同步过，无需再次同步"

### 场景2：多次手动刷新企查查接口

**第一次刷新：**
- 查询所有企业统一社会信用代码（例如：50个）
- 同步数据并存储到 `news_detail` 表

**第二次刷新（同一天）：**
- 查询当天已同步的企业全称
- 通过企业全称映射到统一社会信用代码
- 过滤掉已查询的企业（例如：过滤掉30个）
- 只查询剩余的企业（例如：20个）

## ⚠️ 注意事项

1. **仅手动触发生效**：定时任务不会过滤，会正常查询所有数据
2. **基于当天数据**：只过滤当天的数据，跨天的数据不受影响
3. **时区依赖**：使用 Asia/Shanghai 时区，确保时区配置正确
4. **数据完整性**：如果过滤出错，会继续使用所有数据，不会中断同步
