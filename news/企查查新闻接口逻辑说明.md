# 企查查新闻接口逻辑说明

## 一、触发方式

企查查新闻接口可以通过以下方式触发：

1. **手动触发**：通过前端界面点击"同步"按钮
2. **定时任务**：根据配置的 Cron 表达式自动执行

## 二、完整执行流程

### 步骤1：获取配置信息

1. **获取新闻接口配置** (`news_interface_config` 表)
   - 查找 `interface_type = '企查查'` 且 `is_active = 1` 的配置
   - 如果提供了 `config_id`，则使用指定的配置
   - 否则使用最新的企查查配置

2. **获取企查查应用凭证** (`qichacha_config` 表)
   - 查找 `interface_type = '新闻舆情'` 且 `is_active = 1` 的配置
   - 获取以下信息：
     - `qichacha_app_key`：应用Key
     - `qichacha_secret_key`：应用密钥
     - `qichacha_daily_limit`：每日查询限制次数（默认100次）

### 步骤2：计算时间范围

根据配置的 `frequency_type`（频次类型）计算查询时间范围：

#### 2.1 按周执行（`frequency_type = 'week'`）

**情况A：配置了 `weekday`（星期几执行）**

- **周四执行** (`weekday = 'thursday'`)：
  - 取数周期：**本周一 00:00:00** 到 **本周四 00:00:00**
  - 例如：如果今天是2025-01-16（周四），则获取 2025-01-13 00:00:00 到 2025-01-16 00:00:00 的新闻

- **周一执行** (`weekday = 'monday'`)：
  - 取数周期：**上周四 00:00:00** 到 **本周一 00:00:00**
  - 例如：如果今天是2025-01-13（周一），则获取 2025-01-09 00:00:00 到 2025-01-13 00:00:00 的新闻

- **其他星期几**：
  - 取数周期：**上周一 00:00:00** 到 **上周日 23:59:59**

**情况B：未配置 `weekday`**

- 取数周期：**上周一 00:00:00** 到 **上周日 23:59:59**

#### 2.2 按天执行（`frequency_type = 'day'`）

- 取数周期：**前N天 00:00:00** 到 **今天 00:00:00**
- N = `frequency_value`（默认1天）
- 例如：如果 `frequency_value = 1`，则获取昨天 00:00:00 到今天 00:00:00 的新闻

#### 2.3 按月执行（`frequency_type = 'month'`）

- 取数周期：**上个月第一天 00:00:00** 到 **上个月最后一天 23:59:59**

### 步骤3：获取企业列表

从 `invested_enterprises` 表查询需要同步的企业：

**查询条件：**
- `exit_status NOT IN ('完全退出', '已上市')` - 排除完全退出和已上市的企业
- `unified_credit_code IS NOT NULL` - 统一信用代码不为空
- `unified_credit_code != ''` - 统一信用代码不为空字符串
- `unified_credit_code != 'null'` - 统一信用代码不为字符串'null'
- `delete_mark = 0` - 未删除

**去重处理：**
- SQL层面：使用 `DISTINCT` 去重
- JavaScript层面：再次去重，确保每个统一信用代码只出现一次

### 步骤4：处理每日查询限制

**限制逻辑：**
- 每个企业查询1次企查查接口
- 受 `qichacha_daily_limit`（每日查询限制）限制
- 每次同步最多处理 `dailyLimit` 个企业
- 如果企业总数超过限制，只处理前 `dailyLimit` 个企业
- 剩余企业将在后续同步中处理

**示例：**
- 如果共有150个企业，每日限制100次
- 本次同步处理前100个企业
- 剩余50个企业将在下次同步时处理

### 步骤5：遍历企业并调用接口

对每个企业执行以下操作：

#### 5.1 生成认证Token

```javascript
timestamp = Math.floor(Date.now() / 1000)  // 当前时间戳（秒）
token = MD5(appKey + timestamp + secretKey).toUpperCase()
```

#### 5.2 构建请求参数

```javascript
{
  key: appKey,                    // 应用Key
  searchKey: creditCode,          // 统一信用代码（搜索关键词）
  startDate: startDate,           // 开始日期（YYYY-MM-DD）
  endDate: endDate,               // 结束日期（YYYY-MM-DD）
  pageSize: '50',                 // 每页50条
  pageIndex: '1'                  // 第一页（只查询第一页）
}
```

**注意：**
- 不指定 `emotionType`（情感类型），获取所有情感类型的数据
- 不指定 `category`（类别），获取所有类别的数据

#### 5.3 发送HTTP请求

```javascript
GET https://api.qichacha.com/CompanyNews/SearchNews?key=xxx&searchKey=xxx&...
Headers:
  Token: <生成的Token>
  Timespan: <时间戳>
```

#### 5.4 处理返回数据

**成功响应（Status = '200'）：**

1. **解析返回的新闻列表** (`Result` 或 `result` 字段)

2. **对每条新闻进行去重检查**：
   - 优先使用 `source_url`（Url字段）去重
   - 如果没有 `source_url`，使用 `title` + `public_time` 组合去重
   - 如果已存在，跳过该条新闻

3. **数据转换和存储**：
   - **情感类型转换**：
     - `EmotionType = 'positive'` → `news_sentiment = 'positive'`（正面）
     - `EmotionType = 'negative'` → `news_sentiment = 'negative'`（负面）
     - `EmotionType = 'none'` → `news_sentiment = 'neutral'`（中性）
     - 默认：`news_sentiment = 'neutral'`
   
   - **类别转换**：
     - 将 `Category`（类别代码）转换为中文类别名称
     - 例如：`'1'` → `'企业发展'`，`'2'` → `'荣誉'` 等
   
   - **企业全称匹配**：
     - 根据统一信用代码查找对应的企业全称
     - 从 `invested_enterprises` 表中查询
     - 设置 `enterprise_full_name` 字段

4. **插入数据库前的处理**：
   - 表名：`news_detail`
   - 字段映射：
     ```
     企查查字段 → 数据库字段
     Source → account_name, wechat_account
     Url → source_url
     Title → title
     NewsTags → summary
     PublishTime → public_time
     Content → content（如果为空，则通过AI抓取）
     Category → keywords（JSON格式）
     EmotionType → news_sentiment（转换后）
     Category → news_category（中文类别）
     ```
   - `APItype = '企查查'`
   - `enterprise_full_name`：从 invested_enterprises 表匹配
   
   **AI自动提取正文和摘要（当Content为空时）：**
   
   如果企查查接口返回的 `Content` 字段为空或不存在，系统会自动执行以下操作：
   
   a. **网页内容抓取**：
      - 使用 `source_url`（Url字段）抓取网页HTML内容
      - 设置浏览器User-Agent和请求头，模拟真实浏览器访问
      - 超时时间：30秒
      - 支持最多5次重定向
   
   b. **AI提取正文和生成摘要**：
      - 使用系统中配置的AI大模型（`application_type = 'news_analysis'` 且 `is_active = 1`）
      - 将HTML内容发送给AI模型，要求：
        - 提取正文内容（去除HTML标签、脚本、广告等）
        - 保留段落结构
        - 生成100字左右的关键信息摘要
      - AI返回JSON格式：`{content: "正文内容", abstract: "摘要"}`
   
   c. **更新字段**：
      - 如果AI成功提取正文，更新 `content` 字段
      - 如果AI成功生成摘要，更新 `news_abstract` 字段
      - 如果AI提取失败，`content` 保持为空，不影响数据插入
   
   **AI配置要求**：
   - 需要在"系统配置 → AI模型配置"中配置AI模型
   - 配置要求：`application_type = 'news_analysis'` 且 `is_active = 1`
   - 支持的AI提供商：阿里云（千问）、OpenAI等
   - 系统会自动使用最新创建的启用配置

5. **插入数据库**：
   - 将处理后的数据插入 `news_detail` 表
   - 包含企查查原始数据和AI提取的正文/摘要（如果有）

6. **统计同步数量**：
   - 每成功插入一条新闻，`totalSynced++`

**错误响应：**
- 记录错误信息到 `errors` 数组
- 继续处理下一个企业

#### 5.5 请求延迟

每个企业查询后延迟 **500毫秒**，避免请求过快

### 步骤6：更新同步时间

同步完成后，更新 `news_interface_config` 表：
- `last_sync_time = NOW()`
- `last_sync_date = CURDATE()`

### 步骤7：触发AI分析（如果同步了新数据）

如果 `totalSynced > 0`：

1. **异步启动AI分析**：
   - 调用 `newsAnalysis.batchAnalyzeNews(totalSynced)`
   - 异步执行，不阻塞同步响应
   - 分析内容包括：
     - 新闻情绪分析（正面/中性/负面）
     - 新闻类型分类
     - 新闻摘要生成
     - 企业关联性分析

2. **记录AI分析结果**：
   - 更新 `news_detail` 表中的相关字段

### 步骤8：返回同步结果

返回结果包含：
```javascript
{
  success: true,
  message: "同步完成，共同步 X 条新闻",
  data: {
    synced: totalSynced,           // 成功同步的新闻数量
    total: uniqueCreditCodes.length, // 企业总数
    errors: [...]                   // 错误列表（最多10个）
  }
}
```

## 三、关键特性

### 1. 智能过滤
- **自动排除**：完全退出和已上市的企业不会同步新闻
- **去重机制**：根据 `source_url` 或 `title + public_time` 去重，避免重复数据

### 2. 每日限制处理
- 受每日查询次数限制
- 如果企业数量超过限制，分批处理
- 每次同步处理部分企业，剩余企业后续处理

### 3. 数据完整性
- 获取**所有情感类型**的新闻（正面/中性/负面）
- 获取**所有类别**的新闻（企业发展、荣誉、产品发布等）
- 自动匹配企业全称
- 自动转换类别代码为中文

### 4. AI自动提取正文和摘要
- **自动检测**：当企查查接口返回的 `Content` 为空时，自动触发AI提取
- **网页抓取**：自动抓取 `source_url` 指向的网页内容
- **智能提取**：使用AI大模型提取正文内容，去除HTML标签、广告等无关内容
- **摘要生成**：自动生成100字左右的关键信息摘要
- **配置灵活**：使用系统中配置的AI模型，支持多种AI提供商
- **容错机制**：AI提取失败不影响数据插入，记录错误日志

### 5. 错误处理
- 单个企业失败不影响其他企业
- 记录错误信息，便于排查
- 返回详细的错误列表
- AI提取失败不影响数据插入流程

同步过程中会输出以下日志：

### 4.1 同步统计日志

```
[企查查同步] ========== 同步统计 ==========
[企查查同步] 配置ID: XXX
[企查查同步] 时间范围: YYYY-MM-DD 到 YYYY-MM-DD
[企查查同步] 企业总数: XX
[企查查同步] 本次处理: XX 个企业
[企查查同步] 成功同步: XX 条新闻
[企查查同步] 失败数量: XX
[企查查同步] =============================
```

### 4.2 AI提取日志

当检测到 `Content` 为空时，会输出以下日志：

```
[企查查同步] 检测到content为空，开始通过AI抓取网页内容: https://example.com/news/123
[网页抓取] 开始抓取网页内容: https://example.com/news/123
[网页抓取] 成功抓取网页，内容长度: 15234 字符
[AI提取] 开始使用AI提取正文和生成摘要
[AI提取] 使用的AI配置: alibaba - qwen-turbo (我的AI配置)
[AI提取] 成功提取正文（长度: 1234）和摘要（长度: 98）
[企查查同步] 成功提取正文，长度: 1234 字符
[企查查同步] 成功生成摘要，长度: 98 字符
```

如果AI提取失败，会输出错误日志：

```
[企查查同步] AI提取网页内容失败 (https://example.com/news/123): 抓取网页失败: timeout
```

**注意**：AI提取失败不会中断数据插入流程，数据会以空的 `content` 插入。

## 五、注意事项

1. **时间范围**：根据 `frequency_type` 和 `weekday` 自动计算，不是固定的"昨天"
2. **每日限制**：如果企业数量很多，可能需要多次同步才能处理完所有企业
3. **去重机制**：已存在的新闻不会重复插入
4. **异步AI分析**：AI分析在后台异步执行，不会阻塞同步响应
5. **错误容忍**：单个企业失败不影响整体同步流程
6. **AI提取正文**：
   - 需要先在"系统配置 → AI模型配置"中配置AI模型
   - AI配置要求：`application_type = 'news_analysis'` 且 `is_active = 1`
   - 如果未配置AI模型，`Content` 为空时不会自动提取，数据会以空 `content` 插入
   - AI提取会增加处理时间（每个新闻约30-60秒），建议在非高峰期同步
   - 网页抓取可能受网络、网站反爬虫策略等因素影响，失败时不影响数据插入
7. **性能考虑**：
   - AI提取正文功能会增加同步时间
   - 如果大量新闻需要提取，建议分批处理或使用异步队列
   - 网页抓取超时时间为30秒，超时后会跳过该新闻的提取
