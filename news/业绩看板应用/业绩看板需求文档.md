    1. 定期执行系统取数写入操作，确保数据及时更新，需要配置一个调用页面触发生成数据的接口，定时任务的配置使用当前系统已有的定时任务配置的功能，配置为每月1日和4日凌晨00:00:00触发一次系统创建

DROP PROCEDURE IF EXISTS delete_version_by_condition;

DELIMITER //  

CREATE PROCEDURE delete_version_by_condition()
BEGIN
    -- 声明变量存储关键信息
    DECLARE target_version VARCHAR(50);  -- 最大日期的最大版本号
    DECLARE F_CreatorUserId VARCHAR(50); -- 创建人ID
    DECLARE F_CreatorTime DATETIME;      -- 创建时间
    -- 关键修改1：默认标记为「不删除」，只有满足删除条件时才改为TRUE
    DECLARE delete_flag BOOLEAN DEFAULT FALSE;  

    -- 异常处理：避免无数据/数据异常导致执行失败
    DECLARE EXIT HANDLER FOR SQLEXCEPTION
    BEGIN
        SELECT '执行出错，未删除任何数据' AS result;
    END;

    -- 步骤1：查询b_version中最大日期的最大版本号
    SELECT version INTO target_version
    FROM (
        SELECT 
            version,
            CAST(SUBSTRING(version, 1, 8) AS UNSIGNED) AS date_num,
            CAST(SUBSTRING(version, INSTR(version, 'V') + 1) AS UNSIGNED) AS version_num
        FROM b_version
        WHERE version REGEXP '^[0-9]{8}V[0-9]+$'
    ) t
    ORDER BY date_num DESC, version_num DESC
    LIMIT 1;

    -- 先判断是否查到有效版本，避免空值后续报错
    IF target_version IS NOT NULL THEN
        -- 步骤2：查询该版本对应的创建人ID和创建时间
        SELECT F_CreatorUserId, F_CreatorTime 
        INTO F_CreatorUserId, F_CreatorTime
        FROM b_version
        WHERE version = target_version;

        -- 步骤3：重新设计条件判断逻辑（核心修改2）
        -- 只有「创建人ID为空」且「创建时间不是4日」时，才标记删除
        IF F_CreatorUserId IS NULL THEN
            IF DAY(F_CreatorTime) != 4 AND DAY(F_CreatorTime) != 1 THEN
                SET delete_flag = TRUE;
            END IF;
        END IF;

        -- 步骤4：执行删除或保留逻辑
        IF delete_flag THEN
            -- 删除关联表数据
            DELETE FROM b_version WHERE version = target_version;
            DELETE FROM b_investment_indicator WHERE version = target_version;
            DELETE FROM b_investment_sum WHERE version = target_version;
            DELETE FROM b_investor_list WHERE version = target_version;
            DELETE FROM b_manage_indicator WHERE version = target_version;
            DELETE FROM b_project_all WHERE version = target_version;
            DELETE FROM b_transaction_indicator WHERE version = target_version;
            DELETE FROM b_all_indicator WHERE version = target_version;
            DELETE FROM b_investment WHERE version = target_version;
            DELETE FROM b_ipo WHERE version = target_version;
            DELETE FROM b_manage WHERE version = target_version;
            DELETE FROM b_project WHERE version = target_version;
            DELETE FROM b_transaction WHERE version = target_version;
            DELETE FROM b_project_a WHERE version = target_version;
            DELETE FROM b_region_a WHERE version = target_version;
            DELETE FROM b_region WHERE version = target_version;
            DELETE FROM b_ipo_a WHERE version = target_version;
            
            SELECT CONCAT('已删除版本：', target_version, '（原因：创建人ID为空且创建时间不是1日或4日）') AS result;
        ELSE
            -- 区分保留原因，方便调试
            IF F_CreatorUserId IS NOT NULL THEN
                SELECT CONCAT('版本', target_version, '满足保留条件，未删除（原因：创建人ID非空）') AS result;
            ELSE
                SELECT CONCAT('版本', target_version, '满足保留条件，未删除（原因：创建人ID为空但创建时间是1日或4日）') AS result;
            END IF;
        END IF;
    ELSE
        SELECT '未查询到符合格式的版本数据，无需删除' AS result;
    END IF;

END //

DELIMITER ;  

-- 调用存储过程
CALL delete_version_by_condition();

（三）技术架构规范

1. 系统架构设计

   1.1 整体架构
   - 采用前后端分离架构，前端基于 Arco Design Vue 组件库
   - 后端采用 RESTful API 设计规范，集成到现有新闻舆情系统的 Express 服务中
   - 数据库采用 MySQL 8.0+，使用 InnoDB 存储引擎，复用现有 `db.js` 连接池
   - 无 Redis 依赖，使用 Node.js 进程内 Map 实现热点数据缓存
   - 文件导出使用现有 uploads 目录或直接流式返回

   1.2 技术栈规范（与新闻舆情系统保持一致）
   - 前端：Vue 3 + TypeScript + Arco Design Vue
   - 后端：Node.js 18 + Express 4.x
   - 数据库驱动：mysql2 3.x（直接操作SQL，不使用ORM）
   - 定时任务：node-cron 4.x
   - Excel导出：xlsx 0.18+
   - 参数校验：express-validator 7.x

2. 数据库设计规范

   2.1 连接池配置（复用现有系统配置）
   ```javascript
   // 复用 server/db.js，connectionLimit: 10
   // 如需独立部署可调整至 20
   ```

   2.2 索引设计
   - 所有表必须在 `version` 字段上建立索引
   - 所有表必须在 `F_DeleteMark` 字段上建立索引
   - 组合索引：`(version, F_DeleteMark)`
   - 查询频繁的字段如 `fund`、`b_date` 需单独建立索引

   2.3 事务控制
   - 数据版本更新必须使用数据库事务（通过 `db.getConnection()` 获取连接）
   - 事务隔离级别：默认使用数据库配置（REPEATABLE READ）
   - 失败时必须回滚所有已执行的操作

   2.4 版本号生成并发控制
   ```sql
   -- 使用数据库行锁实现版本号生成的并发控制
   SELECT * FROM b_version WHERE b_date = ? FOR UPDATE;
   ```

   2.5 b_sql 多数据源支持
   - `b_sql` 表新增 `external_db_config_id` 字段（关联 `external_db_config` 表）
   - 值为 NULL 时，SQL 在主库（`db.js` 连接池）执行
   - 值非空时，通过 `utils/externalDb.js` 建立外部数据库连接执行 SQL
   - 新增 `target_table` 字段，指定查询结果写入哪张业务表

   2.6 ID 生成规范
   - 所有业务表主键（`F_Id`）使用 `utils/idGenerator.js` 生成（已添加业绩看板表白名单）
   - 格式：`YYYYMMDDHHmmss + 5位自增序列`（19位，varchar(50)）

3. 接口设计规范

   3.1 统一响应格式（与新闻舆情系统保持一致）
   ```json
   // 成功
   { "success": true, "message": "操作成功", "data": {} }
   // 失败
   { "success": false, "message": "错误原因" }
   ```

   3.2 认证方式（与新闻舆情系统保持一致）
   - 内部用户：请求头携带 `X-User-Id: <userId>`（不使用 JWT）
   - 外部接口（分享页面）：携带 `Authorization: Bearer <api_token>` 或 `X-Api-Token: <api_token>`

   3.3 分页规范
   - 页码从 1 开始
   - 默认每页 20 条
   - 最大每页 100 条
   ```json
   {
     "page": 1,
     "pageSize": 20,
     "total": 100,
     "list": []
   }
   ```

4. 安全规范

   4.1 SQL注入防护
   - 所有用户输入的SQL必须通过参数化查询执行（`db.query(sql, [params])`）
   - SQL配置接口需进行语法解析和危险操作检测（禁止 DROP/DELETE/UPDATE/INSERT）
   - 敏感操作需记录审计日志

   4.2 权限控制
   - 内部接口通过 `X-User-Id` header 识别用户（参考 `middleware/auth.js`）
   - 分享链接使用 `crypto.randomBytes(32).toString('hex')` 生成 token，存入数据库（参考 `newsShare.js`）
   - 权限粒度：查看/编辑/删除/锁定/版本管理

   4.3 数据脱敏
   - 分享页面中投资人名称需脱敏显示（如：上海***公司）
   - 敏感金额数据根据权限级别显示

5. 性能规范

   5.1 响应时间要求
   - 页面首屏加载 < 2s
   - 数据查询接口 < 500ms
   - 弹窗数据加载 < 1s
   - Excel导出（<1万条）< 10s

   5.2 大数据量处理
   - 弹窗表格数据超过 1000 条时必须分页
   - Excel导出采用流式写入，避免内存溢出
   - 基金产品横向滚动采用虚拟滚动（基金数量 > 20）

   5.3 缓存策略（使用 Node.js 内存缓存，不依赖 Redis）
   - 版本列表缓存 5 分钟
   - 指标说明配置缓存 10 分钟
   - 数据生成后清除相关缓存

6. 日志与监控

   6.1 日志规范
   - 所有接口请求记录访问日志（用户、IP、耗时、状态码）
   - 数据版本更新记录操作日志（参考 `utils/logger.js`）
   - SQL执行记录慢查询日志（>1s）

   6.2 监控告警
   - 数据生成任务失败告警
   - 数据库连接池使用率告警（>80%）
   - 接口响应时间告警（>2s）

7. 部署与运维

   7.1 部署架构
   ```
   [Nginx] -> [Express应用服务] -> [MySQL]
   ```
   > 复用现有 docker-compose.yml，不增加新的服务容器

   7.2 备份策略
   - 数据库每日全量备份
   - 导出文件保留 30 天
   - 版本数据软删除，保留 1 年后物理清理

   7.3 高可用
   - 数据库主从复制（可选）
   - 应用与新闻舆情系统合并部署
   - 定时任务单节点执行（使用分布式锁）
