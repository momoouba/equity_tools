# 正常更新部署流程

## 📋 更新前后端代码后的部署步骤

### 场景1: 只更新了后端代码（server目录）

**后端代码通过 volume 挂载，代码更改会自动同步，只需重启容器：**

```bash
cd /opt/newsapp/news

# 1. 重启应用容器（代码已自动同步）
docker compose restart app

# 2. 查看日志确认启动成功
docker compose logs -f app
```

**验证：**
```bash
# 检查容器状态
docker compose ps

# 测试健康检查
curl http://localhost:3001/api/health
```

---

### 场景2: 只更新了前端代码（client目录）

**前端需要重新构建并复制到 volume：**

#### 方法1: 从本地上传构建文件（推荐，最快）

**在本地：**
```powershell
# 1. 构建前端
cd E:\USER\SUREAL\Desktop\equity_news\news\client
npm run build

# 2. 使用 WinSCP 将 dist 目录上传到服务器 /tmp/dist
```

**在服务器：**
```bash
cd /opt/newsapp/news

# 1. 复制到 volume
VOLUME_PATH=$(docker volume inspect news_app_frontend | grep -i "Mountpoint" | awk '{print $2}' | tr -d '",')
sudo rm -rf "$VOLUME_PATH"/*
sudo cp -r /tmp/dist/* "$VOLUME_PATH/"

# 2. 重启 nginx
docker compose restart nginx

# 3. 验证
docker compose exec nginx stat -c%s /usr/share/nginx/html/index.html
```

#### 方法2: 在服务器上构建（推荐，如果服务器有 Node.js）

**使用脚本（推荐）：**
```bash
cd /opt/newsapp/news
chmod +x deploy/在服务器上构建前端.sh
bash deploy/在服务器上构建前端.sh
```

**或手动执行：**
```bash
cd /opt/newsapp/news

# 1. 检查 Node.js（如果没有需要先安装）
node --version || echo "需要安装 Node.js"

# 2. 构建前端
cd client
npm install  # 首次构建需要
npm run build
cd ..

# 3. 复制到 volume
VOLUME_PATH=$(docker volume inspect news_app_frontend | grep -i "Mountpoint" | awk '{print $2}' | tr -d '",')
sudo rm -rf "$VOLUME_PATH"/*
sudo cp -r client/dist/* "$VOLUME_PATH/"

# 4. 重启 nginx
docker compose restart nginx
```

#### 方法3: 在容器内构建（如果服务器没有 Node.js）

```bash
cd /opt/newsapp/news
chmod +x deploy/在容器内构建前端.sh
bash deploy/在容器内构建前端.sh
```

## 📝 重要说明

### 1. 后端代码（server目录）
- ✅ **自动同步**：通过 volume 挂载 `./server:/app/server`
- ✅ **无需重建镜像**：代码更改后只需重启容器
- ✅ **即时生效**：重启后立即生效

### 2. 前端代码（client目录）
- ⚠️ **需要构建**：必须先运行 `npm run build`
- ⚠️ **需要复制到 volume**：构建后的文件需要复制到 `app_frontend` volume
- ⚠️ **需要重启 nginx**：复制后需要重启 nginx 容器

### 3. 数据库变更
- 如果修改了数据库结构，需要手动执行 SQL 迁移
- 或者通过应用启动时的自动创建逻辑（如果实现了）

## 🔨 Docker 完全重新构建流程

### 港交所 iFinD 数据同步说明

**功能说明：**
- 港交所上市申请数据可通过同花顺 iFinD 接口获取
- **Windows 本地环境**：使用本地 iFinDPy SDK + 用户名/密码登录
- **Linux 生产环境**：使用 HTTP API + refresh_token 获取 access_token

**架构对比：**

| 环境 | 认证方式 | 数据获取方式 | SDK 安装 |
|-----|---------|-------------|---------|
| Windows | 用户名+密码 或 refresh_token | 本地 iFinDPy SDK | 手动下载安装 |
| Linux/Docker | refresh_token → access_token | HTTP RESTful API | Dockerfile 自动安装 |

**Windows 环境配置步骤：**
1. 从同花顺官网下载 Windows 版 iFinD SDK
2. 解压到 `THSDataInterface_Windows/` 目录
3. 运行 `python installiFinDPy.py` 安装
4. 打开 iFinD 终端 → 「超级命令」→「工具」→「refresh_token 查询」获取 token
5. 在「系统设置 → 上市数据配置」中填写用户名、密码或 refresh_token

**Linux/Docker 环境配置步骤：**
1. SDK 已在 Dockerfile 中配置自动安装
2. 使用 Windows 环境获取的 refresh_token
3. 在管理后台配置页面填入 Token
4. 系统自动通过 HTTP API 获取数据

**详细文档：**
- [iFinD 集成详细说明](./iFinD集成说明.md)
- [生产环境更新指南 - iFinD 配置](./生产环境更新指南.md#-港交所-ifind-配置说明)

**Linux Docker 环境注意事项：**
1. **确保 Python 依赖已安装**
   ```bash
   # 验证 pandas 是否安装
   docker compose exec app python3 -c "import pandas; print('OK')"
   
   # 如果报错，需要重新构建镜像
   docker compose build --no-cache app
   docker compose up -d
   ```

2. **iFinD SDK 安装**
   - 同花顺 iFinD SDK 需要手动安装，不通过 PyPI 发布
   - Dockerfile 已配置自动安装 Linux 版 SDK
   - Windows 版 SDK 需单独下载，不包含在代码仓库中（已添加到 .gitignore）

3. **状态映射规则**
   | 日期匹配条件 | 系统状态 |
   |-------------|---------|
   | 首次申请日期在区间内 | **递交A1** |
   | 通过聆讯日期在区间内 | **通过聆讯** |
   | 上市日期在区间内 | **上市** |
   | 状态更新日期在区间内 + 终止状态 | 保持原状态 |

4. **常见错误排查**
   ```bash
   # 查看详细错误日志
   docker compose logs app | grep -i ifind
   
   # 常见错误：
   # - ModuleNotFoundError: No module named 'pandas' → 重新构建镜像
   # - 登录失败（返回码 -2）→ refresh_token 过期，需重新获取
   # - 源表行数=0 → 检查日期范围或 refresh_token 有效性
   ```

### 何时需要完全重新构建

以下情况需要执行完全重新构建：
- ✅ 修改了 `Dockerfile`
- ✅ 修改了依赖文件（`package.json`、`package-lock.json`）
- ✅ 修改了前端代码并需要在镜像中构建
- ✅ 添加/删除了系统依赖
- ✅ **修改了 Python 依赖（`requirements.txt`）** ⬅️ 常见！
- ✅ 需要清理旧的镜像和构建缓存
- ✅ 迁移到新环境或全新部署

⚠️ **注意**：如果只是修改了后端业务代码（`server/`目录），不需要重新构建，只需重启容器即可（见场景1）。

⚠️ **Python 依赖特别说明**：
- Python 依赖在 Docker 镜像构建时安装（见 Dockerfile 第 73-79 行）
- 新增 `.py` 文件如果使用了新依赖（如 `pandas`、`akshare`），必须重新构建镜像
- 常见错误：`ModuleNotFoundError: No module named 'pandas'` → 需要重新构建镜像


#### 步骤4: 验证

```bash
docker compose build --no-cache

# 2. 启动所有服务

# 3. 查看日志
```

## 🔍 构建故障排查

### 问题1: 构建失败，提示内存不足

**解决方法：**
```bash
# 增加 Node.js 内存限制
docker compose build --build-arg NODE_OPTIONS="--max-old-space-size=6144"

# 或者增加 Docker 的内存限制（在 Docker Desktop 设置中）
```

### 问题2: 依赖下载失败

**解决方法：**
```bash
# 清理 npm 缓存
docker compose build --no-cache

# 或者修改 Dockerfile 中的镜像源（已配置国内源）
```

### 问题3: 构建速度慢

**解决方法：**
```bash
# 使用多核并行构建
DOCKER_BUILDKIT=1 docker compose build --no-cache --parallel

# 或者只重建有变化的服务
docker compose build --no-cache app
```

### 问题4: 启动后容器立即退出

**解决方法：**
```bash
# 查看详细日志
docker compose logs app

# 进入容器调试
docker compose run --rm app /bin/bash

# 检查环境变量
docker compose exec app env | grep DB_
```

## 🎯 最佳实践

1. **生产环境部署前**
   - 先在测试环境验证
   - 备份数据库
   - 记录当前版本号

2. **构建镜像时**
   - 使用 `--no-cache` 确保使用最新依赖
   - 监控构建日志，及时发现问题
   - 为镜像打标签（便于回滚）

3. **部署时**
   - 使用滚动更新（如果有多台服务器）
   - 保留旧版本镜像至少 24 小时
   - 监控应用日志和性能指标

4. **回滚准备**
   - 保留上一个版本的镜像
   - 准备快速回滚脚本
   - 记录每个版本的变更

## 📝 重新构建 vs 重启容器

| 操作 | 使用场景 | 数据影响 | 耗时 |
|------|---------|---------|------|
| 重启容器 | 修改后端业务代码 | 保留数据 | 快（<1分钟） |
| 更新前端 | 修改前端代码 | 保留数据 | 中等（2-5分钟） |
| 重新构建 | 修改依赖、Dockerfile | 保留数据 | 慢（5-20分钟） |
| 完全重建 | 全新环境、重置 | 删除数据 | 慢（5-20分钟） |
