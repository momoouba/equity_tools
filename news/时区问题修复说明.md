# 时区问题修复说明

## 修复概述

已全面检查并修复系统中所有与时区相关的日期时间计算问题，确保所有时间计算都基于 **Asia/Shanghai** 时区。

## 已修复的问题

### 1. 定时任务执行时的时间范围计算 ✅

**文件：** `server/routes/news.js` - `syncConfigWithSchedule` 函数

**问题：**
- 定时任务执行时，使用 `new Date()` 和 `setDate`/`setHours` 计算日期，可能受服务器时区影响
- `last_sync_time` 解析时可能被当作UTC时间处理

**修复：**
- 使用 `toLocaleDateString` 获取 Asia/Shanghai 时区的本地日期
- 基于本地时区计算 `baseRunDate`、`fromDate`、`toDate`
- 解析 `last_sync_time` 时，将其视为 Asia/Shanghai 时区的时间
- 添加详细的日志输出，便于调试和验证

**影响：**
- ✅ 定时任务执行时的时间范围计算正确
- ✅ 首次执行和后续执行的时间范围都正确

### 2. 手动触发时的时间范围计算 ✅

**文件：** `server/routes/news.js` - 手动同步接口

**问题：**
- 手动触发时，使用 `startOfDay(new Date())` 和 `addDays()` 可能受服务器时区影响
- 传入 `customRange` 的是 Date 对象，而不是字符串

**修复：**
- 使用 `toLocaleDateString` 获取 Asia/Shanghai 时区的本地日期
- 基于本地时区计算昨天和今天的 00:00:00
- 格式化为字符串后传入 `customRange`
- 添加详细的日志输出

**影响：**
- ✅ 手动触发时的时间范围计算正确
- ✅ 不再出现日期偏差问题

### 3. 邮件发送的时间范围计算 ✅

**文件：** 
- `server/utils/scheduledEmailTasks.js` - `getYesterdayTimeRange` 函数
- `server/utils/emailSender.js` - `getYesterdayTimeRange` 函数

**问题：**
- 使用 `new Date()` 和 `setDate`/`setHours` 计算日期，可能受服务器时区影响

**修复：**
- 使用 `toLocaleDateString` 获取 Asia/Shanghai 时区的本地日期
- 基于本地时区计算昨天和今天的 00:00:00

**影响：**
- ✅ 邮件发送时的时间范围计算正确
- ✅ 每日邮件发送的数据范围正确

### 4. 定时任务时间范围计算函数 ✅

**文件：** `server/routes/news.js` - `calculateScheduledSyncTimeRange` 函数

**问题：**
- 使用 `new Date()` 和 `setDate`/`setHours` 计算日期，可能受服务器时区影响

**修复：**
- 使用 `toLocaleDateString` 获取 Asia/Shanghai 时区的本地日期
- 基于本地时区计算前一天的时间范围

**影响：**
- ✅ 定时任务的时间范围计算正确

### 5. 自定义时间范围的解析 ✅

**文件：** `server/routes/news.js` - `syncConfigWithSchedule` 函数

**问题：**
- `new Date(customRange.from)` 可能按 UTC 解析，导致日期偏差

**修复：**
- 添加 `parseLocalDateTime` 函数，将日期字符串解析为本地时区（Asia/Shanghai）的 Date 对象
- 避免 UTC 转换导致的日期偏差

**影响：**
- ✅ 自定义时间范围的解析正确

## 时区配置

### Docker 环境

**文件：** `docker-compose.yml`

```yaml
environment:
  TZ: Asia/Shanghai
```

**说明：**
- Docker 容器已设置时区为 `Asia/Shanghai`
- 确保容器内的系统时区正确

### Cron 定时任务

**文件：** `server/index.js`、`server/utils/scheduledEmailTasks.js`、`server/utils/enterpriseSyncTasks.js`

```javascript
cron.schedule('0 0 * * *', async () => {
  // ...
}, {
  scheduled: true,
  timezone: 'Asia/Shanghai'
});
```

**说明：**
- 所有 cron 定时任务都已设置 `timezone: 'Asia/Shanghai'`
- 确保定时任务在正确的时间触发

## 修复后的行为

### 定时任务执行

**执行时间：** 每天 00:00:00 (Asia/Shanghai)

**时间范围：**
- 首次执行：前一天 00:00:00 到 当天 00:00:00
- 后续执行：上次同步时间 到 当天 00:00:00

**示例：**
- 如果今天是 2025-12-04，定时任务在 00:00:00 触发
- 首次执行：同步 2025-12-03 00:00:00 到 2025-12-04 00:00:00 的数据
- 第二次执行：同步 2025-12-04 00:00:00 到 2025-12-05 00:00:00 的数据

### 手动触发

**时间范围：** 前一天 00:00:00 到 当天 00:00:00

**示例：**
- 如果今天是 2025-12-04，手动触发时
- 同步 2025-12-03 00:00:00 到 2025-12-04 00:00:00 的数据

### 邮件发送

**时间范围：** 前一天 00:00:00 到 当天 00:00:00

**示例：**
- 如果今天是 2025-12-04，邮件发送时
- 包含 2025-12-03 00:00:00 到 2025-12-04 00:00:00 的新闻数据
