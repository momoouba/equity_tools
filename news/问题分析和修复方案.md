# 新闻同步问题分析和修复方案

## 问题1：手动触发同步时间范围不正确

### 问题描述
今天下午17:00左右点击了新榜的新闻更新的手动触发接口，但是同步的数据是11月24日至12月2日的，而不是预期的"昨天"的数据。

### 问题原因

在 `server/routes/news.js` 的 `syncConfigWithSchedule` 函数中（第594-596行），存在以下逻辑：

```javascript
if (config.last_sync_time) {
  // 如果有上一次同步时间，先尝试使用上一次同步时间到当天的范围
  fromDate = new Date(config.last_sync_time);
}
```

**问题分析：**
1. 手动触发时，虽然传入了 `customRange`（前一天00:00:00到今天00:00:00）
2. 但在 `syncConfigWithSchedule` 函数中，如果没有启用 `customRangeEnabled`，会优先使用 `last_sync_time`
3. 如果配置的 `last_sync_time` 是 11月24日，那么手动触发时会从 11月24日 同步到今天（12月4日），导致同步了大量历史数据

### 修复方案

修改 `syncConfigWithSchedule` 函数，确保手动触发时优先使用传入的 `customRange`，而不是 `last_sync_time`：

```javascript
// 修改前（第593-600行）
toDate = new Date(baseRunDate);
if (config.last_sync_time) {
  // 如果有上一次同步时间，先尝试使用上一次同步时间到当天的范围
  fromDate = new Date(config.last_sync_time);
} else {
  // 如果没有上一次同步时间，直接使用前一天00:00:00到当天00:00:00
  fromDate = startOfDay(addDays(toDate, -1));
}

// 修改后
toDate = new Date(baseRunDate);
if (isManual) {
  // 手动触发时，优先使用前一天00:00:00到当天00:00:00
  fromDate = startOfDay(addDays(toDate, -1));
} else if (config.last_sync_time) {
  // 定时任务时，如果有上一次同步时间，使用上一次同步时间到当天的范围
  fromDate = new Date(config.last_sync_time);
} else {
  // 如果没有上一次同步时间，直接使用前一天00:00:00到当天00:00:00
  fromDate = startOfDay(addDays(toDate, -1));
}
```

## 问题2：11月23日之前的数据丢失

### 问题描述
数据库中11月23日之前的数据没有了。

### 可能的原因

1. **数据库表结构迁移**：在 `server/db.js` 的 `migrateTableIdField` 函数中（第76行），如果表结构需要迁移，可能会清空表数据：
   ```javascript
   console.warn(`⚠️  自动迁移模式：将清空表 ${tableName} 的所有数据并重建表结构！`);
   ```

2. **手动删除**：管理员可能通过界面手动删除了数据

3. **数据库操作**：可能有人直接在数据库中执行了删除操作

4. **查询过滤**：数据可能还在，但查询时被过滤掉了（例如 `delete_mark = 1`）

### 排查步骤

1. **检查数据是否真的被删除**
   ```sql
   -- 检查所有数据（包括已删除的）
   SELECT COUNT(*) FROM news_detail;
   
   -- 检查未删除的数据
   SELECT COUNT(*) FROM news_detail WHERE delete_mark = 0;
   
   -- 检查已删除的数据
   SELECT COUNT(*) FROM news_detail WHERE delete_mark = 1;
   
   -- 检查11月23日之前的数据
   SELECT COUNT(*) FROM news_detail 
   WHERE public_time < '2024-11-23 00:00:00' AND delete_mark = 0;
   ```

2. **检查数据库日志**
   ```bash
   # 查看MySQL的binlog（如果启用了）
   sudo docker compose exec mysql mysqlbinlog /var/lib/mysql/binlog.000001 | grep -i "delete\|truncate\|drop"
   ```

3. **检查应用日志**
   ```bash
   # 查看是否有表迁移的日志
   sudo docker compose logs app | grep -i "迁移\|清空\|truncate\|drop"
   ```

### 数据恢复方案

如果数据确实被删除了，可以尝试：

1. **从备份恢复**（如果有备份）
   ```bash
   # 恢复数据库备份
   sudo docker compose exec mysql mysql -u newsapp -pNewsApp@2024 investment_tools < backup.sql
   ```

2. **检查逻辑删除的数据**
   ```sql
   -- 恢复逻辑删除的数据
   UPDATE news_detail 
   SET delete_mark = 0, delete_user_id = NULL, delete_time = NULL 
   WHERE delete_mark = 1 AND public_time < '2024-11-23 00:00:00';
   ```

3. **重新同步历史数据**（如果接口支持）
   - 手动触发同步，指定时间范围为11月23日之前

## 修复代码

需要修改 `server/routes/news.js` 文件，确保手动触发时使用正确的时间范围。
