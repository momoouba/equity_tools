# 生产环境部署说明（使本次修改生效）

本文档说明如何将**前后端修改**在生产环境（news.gf-dsai.com）生效。  
本次变更**不需要修改 Nginx 配置**，仅需更新代码、重新构建并重启服务。

---

## 一、部署方式概览

项目支持两种常见部署方式，任选其一执行即可。

| 方式 | 适用场景 | 主要步骤 |
|------|----------|----------|
| **Docker** | 当前使用 docker-compose（推荐） | 拉代码 → 构建镜像 → 重启容器 |
| **非 Docker** | 服务器直接跑 Node + PM2，Nginx 单独安装 | 拉代码 → 装依赖 → 构建前端 → 重启 PM2 →（可选）重载 Nginx |

---

## 二、方式 A：Docker 部署（推荐）

假设项目目录为 `/opt/newsapp`（或你实际路径），且使用 `news/docker-compose.yml`。

### 1. 进入项目目录

```bash
cd /opt/newsapp
# 或你的实际路径，例如：cd /path/to/equity_news/news
```

### 2. 拉取最新代码

```bash
git fetch origin
git pull origin main
# 若主分支名为 master 或 sync-issue-5849a，请替换 main
```

### 3. 重新构建并启动（含前端构建 + 后端）

```bash
docker compose build --no-cache
docker compose up -d
```

或使用旧版命令：

```bash
docker-compose build --no-cache
docker-compose up -d
```

说明：

- `build --no-cache` 会重新构建镜像，前端（Vite）和后端（Node）都会用最新代码。
- Dockerfile 已包含前端构建阶段，无需单独在宿主机执行 `npm run build`。
- 数据库表结构（如 `users.api_token`、`external_db_config` 的 `created_by` 等）在应用**首次启动**时由 `db.js` 自动迁移，无需手动执行 SQL。

### 4. 查看服务状态

```bash
docker compose ps
docker compose logs -f app
```

确认无报错、健康检查通过即可。

### 5. 若前端未更新：查看 Nginx 使用的 volumes 并删除卷后重启

**为什么会出现“代码已改但页面还是旧的”？**  
前端静态文件由 **命名卷 `app_frontend`** 提供：应用容器把构建好的 `dist` 写入该卷，Nginx 从同一卷读取并对外提供。卷一旦创建会一直保留**旧内容**，仅做 `build` + `up -d` 不会把新镜像里的 `dist` 写进已有卷，所以 Nginx 仍在发旧的前端文件。

**如何查看 Nginx（及应用）使用的 volumes：**

- 看 **docker-compose 配置**（当前项目）：
  - `news/docker-compose.yml` 里：
    - **app**：`volumes: - app_frontend:/app/client/dist`
    - **nginx**：`volumes: - app_frontend:/usr/share/nginx/html:ro`
  - 底部 `volumes:` 定义了命名卷：`app_frontend`、`mysql_data`。
- 在服务器上**列出容器和卷**：
  ```bash
  cd /opt/newsapp
  docker compose ps
  docker volume ls | grep newsapp
  # 或
  docker volume ls
  ```
  卷名一般为 `newsapp_app_frontend` 或 `<项目目录名>_app_frontend`（以实际 `docker volume ls` 为准）。

**推荐做法：先删卷，再重新构建并启动，让 Nginx 使用新生成的前端文件**

1. 进入项目目录并停止容器，**只删除前端卷**（不要用 `docker compose down -v`，否则会连数据库卷 `mysql_data` 一起删掉）：
   ```bash
   cd /opt/newsapp
   docker compose down
   docker volume ls
   # 找到名为 xxx_app_frontend 的卷，再删除
   docker volume rm newsapp_app_frontend
   ```
   若提示卷名不存在，用上一步 `docker volume ls` 查到的名字替换 `newsapp_app_frontend`（可能与项目目录名有关，如 `news_app_frontend`）。
2. 拉取最新代码并重新构建镜像（镜像内会包含新前端）：
   ```bash
   git pull origin main
   docker compose build --no-cache
   ```
3. 启动服务（会新建空卷，并由 **app** 容器首次挂载时从新镜像中复制 `/app/client/dist` 到卷中，Nginx 即可读到新文件）：
   ```bash
   docker compose up -d
   ```
4. 查看 Nginx 是否在提供新前端（可选）：
   ```bash
   docker compose exec nginx ls -la /usr/share/nginx/html
   docker compose exec nginx cat /usr/share/nginx/html/index.html | head -5
   ```

之后让用户**强刷页面**（Ctrl+Shift+R）或清缓存后再访问。

## 三、方式 B：非 Docker 部署（PM2 + 本机 Nginx）

适用于后端用 PM2 管理、Nginx 单独安装在同一台机上的情况。

### 1. 进入项目目录

```bash
cd /opt/newsapp/news
# 或你的实际应用路径（与 ecosystem.config.js 中 cwd 一致）
```

### 2. 拉取最新代码

```bash
git fetch origin
git pull origin main
```

### 3. 后端：安装依赖并重启 PM2

```bash
npm install --production
pm2 reload ecosystem.config.js --env production
```

若未使用 ecosystem 配置文件，可直接重启应用名：

```bash
pm2 reload newsapp
```

查看状态与日志：

```bash
pm2 status
pm2 logs newsapp
```

### 4. 前端：构建并部署到 Nginx 静态目录

在**同一台服务器**上构建前端，并把产物放到 Nginx 配置中的 `root` 目录（例如 `/usr/share/nginx/html`）。

```bash
cd /opt/newsapp/news/client
npm install
npm run build
```

若内存不足可改用：

```bash
npm run build:low-memory
```

将构建产物复制到 Nginx 使用的根目录（路径以你实际 Nginx 配置为准）：

```bash
sudo cp -r dist/* /usr/share/nginx/html/
```

若 Nginx 使用其他路径（例如 `/var/www/news.gf-dsai.com`），请替换为实际路径：

```bash
sudo cp -r dist/* /var/www/news.gf-dsai.com/
```

### 5. Nginx：仅在修改过配置时重载

**本次代码修改未改动 Nginx 配置**，一般不需要执行。若你单独改过 Nginx 配置，再执行：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

## 四、Nginx 说明（本次无需改配置）

当前 `deploy/nginx-site.conf` 已具备：

- HTTP 80 → HTTPS 301
- HTTPS 443、SSL 证书、`news.gf-dsai.com`
- `/api` 反向代理到后端（如 `app:3001` 或 `127.0.0.1:3001`）
- 前端静态资源与 SPA 回退到 `index.html`

本次需求（数据库连接权限、定时更新按用户、news-detail 接口与鉴权）**不涉及** Nginx 修改，因此：

- **Docker 部署**：无需改 Nginx 配置，无需单独重载 Nginx（随容器重启即可）。
- **非 Docker 部署**：仅在你改了 Nginx 配置后才需要执行 `nginx -t` 与 `systemctl reload nginx`。

若将来要改域名、API 前缀或静态路径，再编辑对应 server 块并重载 Nginx。

## 五、部署后自检

在服务器或本机执行以下检查（将域名替换为你的生产域名）。

### 1. 健康检查

```bash
curl -s https://news.gf-dsai.com/api/health
# 或
curl -s https://news.gf-dsai.com/health
```

预期返回包含 `"success":true` 或 `"status":"ok"` 的 JSON。

### 2. 登录并检查 API Token（注册/登录即写 user 表）

```bash
curl -s -X POST https://news.gf-dsai.com/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"account":"admin","password":"你的密码"}'
```

响应中应包含 `user.id` 及 `user.api_token`（若为本次修改后的登录逻辑）。

### 3. 新闻详情接口鉴权

```bash
TOKEN="上一步返回的 api_token"
curl -s "https://news.gf-dsai.com/api/news-detail?page=1&pageSize=5" \
  -H "Authorization: Bearer $TOKEN"
```

预期：HTTP 200，且 JSON 中 `success: true`、有 `data` 数组。

### 4. 前端功能

- 使用**普通用户**（有舆情权限）登录，侧边栏应出现「系统配置」，进入后仅显示「数据库连接」Tab。
- 在「数据库连接」中新增一条配置，列表仅显示自己创建的；管理员应能看到全部。
- 在「舆情监控对象」中打开「定时更新」，选择数据库下拉：普通用户仅能看到自己创建的库；保存的 SQL 按用户隔离。

## 六、常用命令速查

| 操作 | Docker | 非 Docker |
|------|--------|-----------|
| 拉代码 | `git pull origin main` | 同左 |
| 后端重启 | `docker compose up -d`（或先 build） | `pm2 reload newsapp` |
| 前端构建 | 已包含在 `docker compose build` | `cd client && npm run build` |
| 前端部署 | 无需单独操作（镜像内已含 dist） | `cp -r client/dist/* /usr/share/nginx/html/` |
| Nginx 重载 | 一般不需要 | `sudo nginx -t && sudo systemctl reload nginx` |
| 看日志 | `docker compose logs -f app` | `pm2 logs newsapp` |

按上述步骤执行后，本次前后端修改（含数据库连接权限、定时更新按用户、news-detail 接口与鉴权等）即可在生产环境生效。
